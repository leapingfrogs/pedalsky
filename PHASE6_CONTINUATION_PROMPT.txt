HARD-WON GUIDANCE — DO NOT REPEAT THESE MISTAKES
================================================================================

These came up in earlier phases and will absolutely re-emerge for clouds.

────────────────────────────────────────────────────────────────────────────────
1. PLANET-SCALE FP32 CANCELLATION  (single biggest gotcha)
────────────────────────────────────────────────────────────────────────────────

The expression `length(p) - planet_radius_m` is BROKEN for any march sample
point `p` near the planet surface.  `|p|² ≈ R² ≈ 4e13` dominates the
small height offset; squaring loses ~8 decimal digits of fp32 precision,
the small h disappears, and the bake silently returns the **initial**
altitude for every step of the march.  This bug produced ~10× over-bright
multi-scatter values and NaN-filled aerial-perspective froxels in Phase 5
before being diagnosed.

Use the algebraic identity instead.  For a ray `pi = p0 + view_dir·t`:

    |pi|² = r0² + 2·t·r0·cos_view + t²
    |pi| − r0 = (2·t·r0·cos_view + t²) / (r0 + |pi|)

In WGSL:

    let r0 = length(p0);                           // do this ONCE outside the loop
    let cos_view = dot(p0 / max(r0, 1.0), view_dir);
    let h0 = r0 - world.planet_radius_m;
    // inside the loop:
    let pi = p0 + view_dir * t;
    let r_delta_num = 2.0 * t * r0 * cos_view + t * t;
    let r_delta = r_delta_num / (max(r0, 1.0) + length(pi));
    let h = h0 + r_delta;

NB: the `r_delta_num` must include the `* r0` factor.  An earlier attempt
omitted it (used `2t·cos_view + t²/r0` instead of `2t·r0·cos_view + t²`),
which produced r_delta ~6 million times too small — h stuck at the
initial value and every step's extinction was wrong by a constant factor.

The Phase 5 cloud march helpers `compute_altitude_m(p_world)` in the plan
text use `length(p - planet_center) - planet_radius_m` directly.  At
typical cloud altitudes (1.5–18 km) the cancellation is less brutal than
at sea level, but you should STILL switch to the stable form whenever
you're doing a per-step march.  Either:
  (a) compute h once per ray entry (h0 at t0) and accumulate r_delta along
      the march, or
  (b) work in km internally (r0 ≈ 6360 instead of 6_360_000) so squaring
      doesn't lose precision.

────────────────────────────────────────────────────────────────────────────────
2. VIEW DIRECTION FROM INFINITE-FAR REVERSE-Z PROJECTION
────────────────────────────────────────────────────────────────────────────────

`normalize(far_p - near_p)` reconstructed from inv_view_proj is WRONG for
an infinite-far perspective.  At NDC z=0 (far plane) w → 0; the perspective
divide produces NaN or infinity, and `normalize` returns garbage like
(0, ±1, 0).  This produced a uniformly black sky in Phase 5 until traced.

Use the near plane only, subtract camera position:

    let near_h = frame.inv_view_proj * vec4<f32>(ndc.x, ndc.y, 1.0, 1.0);
    let near_p = near_h.xyz / near_h.w;
    let view_dir = normalize(near_p - frame.camera_position_world.xyz);

(Mind the NDC.y sign convention — in PedalSky the vertex shader emits
`p = vec2(x*2-1, 1 - y*2)`, so +Y in NDC is up.  In the AP bake we use
`-ndc.y` because gid.y=0 is the *top* of the framebuffer.)

────────────────────────────────────────────────────────────────────────────────
3. STORAGE-WRITE vs SAMPLEABLE BIND-GROUP CONFLICT
────────────────────────────────────────────────────────────────────────────────

wgpu won't let a texture be bound as `storage<write>` in one binding *and*
sampled via `texture_2d<f32>` in another binding in the same pass.  For
multi-pass bakes that both write LUT N and sample LUT N−1, define
separate bind-group layouts (e.g. `atmosphere_transmittance_only_bind_group_layout`,
`atmosphere_static_only_bind_group_layout`) so each bake sees only the
LUTs it needs to sample.  See `crates/ps-atmosphere/src/lib.rs` for the
pattern.

For the noise volumes in Phase 6 this is more relaxed (each volume is
written once at startup, then sampled forever), but if you do a chained
"base noise → detail-modulated thing" bake, watch for it.

────────────────────────────────────────────────────────────────────────────────
4. F16 RANGE / PRE-MULTIPLIED ILLUMINANCE IN LUTS
────────────────────────────────────────────────────────────────────────────────

f16 max is 65504.  Pre-multiplying a per-unit-illuminance LUT by
`sun_illuminance` (~130 000 lux at noon) immediately saturates and the
LUT loses spectral character — the sky goes grey because RGB channels
all clip at the same max.

Default: bake LUTs in *per-unit-illuminance* and multiply at sample
time in the consuming shader.  Phase 5 sky-view LUT does this.  The AP
LUT currently bakes in sun_illuminance — that's a separate convention
(luminance values fit in f16 for typical 32 km froxel paths) and works,
but document the convention near the textureStore.

────────────────────────────────────────────────────────────────────────────────
5. WGSL RESERVED KEYWORDS
────────────────────────────────────────────────────────────────────────────────

`layout` is a reserved WGSL keyword.  Don't use it as a struct field
or local-variable name; rename to `tile_layout` or similar.  The naga
parser was tolerant in some versions and strict in others.

────────────────────────────────────────────────────────────────────────────────
6. STD140 LAYOUT DRIFT
────────────────────────────────────────────────────────────────────────────────

WGSL uniform structs follow std140-ish layout.  vec3 has 4-element
alignment; trailing scalars often need explicit padding.  Phase 4 has
a naga cross-check that catches mismatches between the Rust `#[repr(C)]`
and the WGSL struct — keep using it.  If you add a CloudParams uniform
buffer for Phase 6, cross-check on first compile.

────────────────────────────────────────────────────────────────────────────────
7. TEST CAMERA AT WORLD Y=0
────────────────────────────────────────────────────────────────────────────────

Placing the camera at world `(0, 0, 0)` puts it exactly on the planet
surface in atmosphere-local coordinates and produces degenerate rays
(many bakes return early because `t_max <= 0`).  Default test camera
must sit at +1.7 m (eye level).  HeadlessApp::render_one_frame_with(camera)
exists for tests that need a specific pose (e.g. pitch=60° to look at the
upper hemisphere for blue-sky validation).

────────────────────────────────────────────────────────────────────────────────
8. EXPOSURE / EV100 AND THE "ZERO" TRAP
────────────────────────────────────────────────────────────────────────────────

ACES tone-map at EV100=15 maps ~10 000 cd/m² to mid-grey.  If a test
shows avg RGB = [0, 0, 0], that does NOT necessarily mean the pass
isn't running — it can mean the scene is below the exposure threshold.
Confirm pass execution by writing a known bright constant (e.g.
`vec4(100000, 0, 0, 1)`) before assuming the shader didn't run.

────────────────────────────────────────────────────────────────────────────────
9. BLUE-NOISE JITTER MUST BE SPATIAL-ONLY
────────────────────────────────────────────────────────────────────────────────

For the cloud march start jitter: read the blue-noise texture by
`vec2<i32>(frag_coord.xy) % 64` ONLY — do NOT add a time-varying
component.  Phase determinism requires that pausing the simulation
produces a frozen image (no shimmer).  This is in the top-level
principles (#9) and the plan §6.
